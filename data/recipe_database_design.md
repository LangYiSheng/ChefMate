# 菜谱数据库设计文档

## 1. 设计目标

当前阶段只准备菜谱数据，因此数据库设计要满足三件事：

1. 能存下菜谱本身的核心信息
2. 能支持后续推荐、备菜检查和烹饪指导
3. 标签集合固定，便于后续做结构化推荐

## 2. 一个菜谱至少要有哪些信息

如果只保存“菜品名、菜品描述、菜品标签、菜品步骤”，对于后续功能来说还不够。

最少建议保留以下 6 类信息：

### 2.1 基础信息

- 菜谱 ID
- 菜品名
- 菜品图片路径
- 菜品描述
- 难度
- 预计烹饪时长
- 默认份量

### 2.2 食材信息

- 食材名
- 参考数量
- 单位
- 是否可选
- 用途说明

说明：

- 没有食材信息，后面就没法做“已有食材推荐”“缺料判断”“动态调整份量”

### 2.3 步骤信息

- 步骤序号
- 步骤内容
- 是否有建议计时
- 额外提醒

说明：

- 步骤不能只存成一整段文本，最好拆成多条步骤
- 这样后面前端才能逐步展示，也方便自动计时提醒

### 2.4 标签信息

- 固定标签集合中的多个标签

说明：

- 标签不是自由输入，而是从固定标签字典中选择
- 这样后面推荐模块和统计模块才好做

### 2.5 厨具信息

- 该菜谱通常需要哪些厨具

说明：

- 这部分放在 `tags.tool` 中统一管理，不再单独拆成外层字段

### 2.6 补充提示信息

- 做菜提示
- 常见翻车点

说明：

- 这部分虽然不是强制，但对烹饪指导很有帮助

## 3. 当前推荐的菜谱数据范围

因此，当前阶段建议一个菜谱至少包含以下字段：

| 类别 | 字段 |
| --- | --- |
| 基础信息 | `name` `image_path` `description` `difficulty` `estimated_minutes` `servings` |
| 食材信息 | `ingredients` |
| 步骤信息 | `steps` |
| 标签信息 | `tags` |
| 额外提示 | `tips` |

结论：

- 不是只有“菜品名、菜品描述、菜品标签、菜品步骤”就够了
- 至少还要补上“食材信息、时长、难度、份量、厨具标签”

## 4. 标签应该怎么设计

你问的核心点其实是：

- 标签集合要不要固定
- 数据库存“010101”好，还是直接存标签更好

我的建议是：

**固定总标签集合，但数据库主存储先保存“标签字典 + 菜谱标签关联”；0101 向量放到推荐侧生成，不直接硬塞进菜谱主表。**

## 5. 为什么不建议一上来就把菜谱主表设计成很多 0/1 字段

如果直接在 `recipe` 表里写成：

- `is_spicy`
- `is_low_oil`
- `is_quick`
- `is_stir_fry`
- `is_rice_friendly`

这种很多布尔字段的形式，会有几个问题：

1. 标签一多，主表会越来越臃肿
2. 新增标签时需要改表结构
3. 调试和维护起来不够灵活
4. 分类标签和多选标签混在一起后会很乱

## 6. 为什么也不建议只存自由文本标签

如果菜谱直接带一组自由文本标签，例如：

- `["清淡", "快手", "下饭"]`

虽然看起来简单，但也有问题：

1. 容易出现同义词不统一
2. 后续很难做稳定过滤和统计
3. 不利于推荐模块生成稳定特征

## 7. 当前最合适的方案

当前阶段最适合的方案是三层结构：

### 7.1 固定标签字典

单独维护一张标签字典表，统一规定：

- 标签分类
- 标签编码
- 标签名称
- 固定排序
- 固定 bit 位

### 7.2 菜谱与标签关联

每个菜谱通过关联表挂上若干标签。

这样做的好处是：

- 人可读
- 好维护
- 标签总集合固定
- 后续可以很方便生成结构化特征

### 7.3 推荐侧生成 0/1 特征向量

当推荐模块需要做打分、召回或模型训练时，再根据固定标签字典把菜谱映射成 0/1 向量。

也就是说：

- **数据库里主存标签**
- **推荐侧生成 bitmap / one-hot / multi-hot**

这样比直接把 bitmap 作为主存储更稳妥。

## 9. 为什么暂时不建议把“所有食材都做成固定标签”

食材种类非常多，而且后面还会不断扩展。

如果一开始就把所有食材都放进固定标签：

- 标签集合会膨胀得非常快
- 维护成本高
- 还容易和真正的食材明细表重复

所以当前阶段建议：

- **食材本身单独存在 `recipe_ingredient` 表**
- **标签只保留适合做推荐过滤的高层特征**

## 10. 当前数据库设计结论

当前阶段建议创建以下几张核心表：

1. `recipe`
2. `recipe_ingredient`
3. `recipe_step`
4. `recipe_tag_category`
5. `recipe_tag`
6. `recipe_tag_map`

## 11. 推荐的数据库存储策略

### 11.1 主存储策略

- `recipe` 表保存菜谱基础信息
- `recipe_ingredient` 表保存食材明细
- `recipe_step` 表保存步骤明细
- `recipe_tag_category` 和 `recipe_tag` 保存固定标签字典
- `recipe_tag_map` 保存菜谱与标签的对应关系

### 11.2 推荐侧特征策略

后续推荐模块如果需要 010101 形式特征，可以这样做：

1. 读取固定标签字典
2. 按分类和 bit 位生成固定长度向量
3. 将菜谱已挂载标签映射成 multi-hot 特征

这一步不必强行放在主数据库层完成。

## 12. 这次 SQL 里会建哪些表

这次初版 SQL 会创建：

- `recipe`
- `recipe_ingredient`
- `recipe_step`
- `recipe_tag_category`
- `recipe_tag`
- `recipe_tag_map`

并预置一套固定标签字典，作为后续录入菜谱数据的基础。

## 13. 面向 AI 提取程序的结构化 JSON 建议

后续如果让 AI 将原始菜谱 Markdown 转为结构化数据，建议统一输出以下 JSON 结构：

```json
{
  "name": "西红柿炒鸡蛋",
  "image_path": null,
  "description": "经典家常快手菜，酸甜开胃，适合新手。",
  "difficulty": "简单",
  "estimated_minutes": 15,
  "servings": 2,
  "tags": {
    "flavor": ["酸甜", "清淡"],
    "method": ["炒"],
    "scene": ["家常", "快手", "下饭"],
    "health": ["低油"],
    "time": ["10到20分钟"],
    "tool": ["炒锅"]
  },
  "ingredients": [
    {
      "name": "西红柿",
      "amount_value": 2,
      "amount_text": "2个",
      "unit": "个",
      "optional": false,
      "purpose": "主体风味"
    }
  ],
  "steps": [
    {
      "step_no": 1,
      "title": "处理食材",
      "instruction": "西红柿洗净切块，鸡蛋打散。",
      "timer_seconds": null,
      "notes": null
    }
  ],
  "tips": [
    "西红柿先炒出汁再回鸡蛋，口感更好。"
  ]
}
```

建议原则：

- `image_path` 暂时允许为空，后续有图片时再补
- `tags` 只能从固定标签集合中选择，不能让模型自由造词
- `difficulty` 只保留外层字段，不再放入 `tags`
- `time` 原则上只保留一个值
- `timer_seconds` 只有原文明确存在计时时再填写
