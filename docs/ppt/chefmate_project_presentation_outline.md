# ChefMate 项目交流展示 PPT 结构与内容大纲

## 0. 使用建议

- 建议页数：12 页正文 + 1 页 Q&A。
- 建议时长：8-10 分钟，其中 PPT 讲解 6-7 分钟，系统演示 2-3 分钟。
- 叙事主线：开场 -> 问题 -> 洞察 -> 方案 -> 效果 -> 演示 -> 收获。
- 展示口径：把 ChefMate 定位为“面向一次完整做饭任务的智能烹饪助手”，不要只讲成“菜谱查询系统”。
- 演示节奏：PPT 中只解释系统能力边界和技术成果，具体功能使用放到应用系统中现场演示。

## 1. 整体视觉风格建议

- 风格关键词：现代、简洁、可信、生活化、有技术感。
- 主色建议：清爽绿色 `#22C55E` 表达食材与健康，深灰 `#111827` 表达专业，青色 `#06B6D4` 表达智能链路，少量珊瑚橙 `#F97316` 用于强调风险、缺料、待处理状态。
- 背景建议：以白色或极浅灰为主，避免大面积暗色背景；关键图表可用浅色分区。
- 字体建议：标题 32-40pt，正文 20-24pt，图表标注 14-16pt；每页最多 3-5 个要点。
- 版式建议：多用“左侧结论 + 右侧图示”的结构；系统架构页使用横向分层图；效果页使用指标卡和条形图。
- 图标建议：用户、对话、图片、语音、菜谱、任务状态、数据库、大模型、语音识别、部署等使用线性图标，保持统一风格。

## Slide 1. ChefMate 智能烹饪助手

**页面目的**

用一句话建立项目定位，让听众一开始就知道 ChefMate 解决的是“做饭任务全过程”。

**页面核心思想**

ChefMate 是一个从“今天吃什么”陪到“怎么一步步做出来”的智能烹饪助手。

**建议页面内容**

标题：

> ChefMate 智能烹饪助手

副标题：

> 面向日常做饭场景的对话式任务协作系统

三句话定位：

- 不只展示菜谱，而是陪用户完成一次完整做饭任务
- 支持自然语言对话、菜品推荐、食材识别、备料检查、步骤指导
- 用智能体工作流把“想法 -> 选择 -> 准备 -> 烹饪”串成闭环

底部信息：

- 项目交流展示
- 汇报人 / 小组 / 日期

**视觉设计指导**

- 使用大标题 + 一张厨房台面或食材照片作为右侧主视觉。
- 左侧放项目名称与一句话价值主张，右侧放流程小标签：想吃什么、食材够吗、缺什么、下一步。
- 可使用一条细线流程：需求表达 -> 推荐 -> 备料 -> 烹饪。

**讲述重点**

开场不要急着讲技术栈，先强调项目选题来自真实做饭流程中的连续困难：很多系统能给菜谱，但 ChefMate 更关注“用户能不能顺利把饭做出来”。

## Slide 2. 今天汇报什么

**页面目的**

告诉听众整场展示的逻辑，降低后续信息密度带来的理解成本。

**页面核心思想**

汇报按“为什么做、怎么做、效果如何、系统怎么用”展开。

**建议页面内容**

标题：

> 今天汇报什么

四个部分：

- 项目背景：为什么选择智能烹饪助手
- 智能方法：阶段型对话 Agent 的任务推进路线与效果
- 系统实现：前后端技术栈、数据与接口设计
- 系统演示：从推荐到备料再到烹饪指导的完整流程

一句过渡：

> 先从用户真实做饭时会遇到的问题开始。

**视觉设计指导**

- 使用 4 段横向导航条，每段配一个线性图标。
- 当前页可以用“汇报路线图”形式，不需要放太多文字。
- 可在底部放一条叙事轴：开场 -> 问题 -> 洞察 -> 方案 -> 收获。

**讲述重点**

说明 PPT 不会逐个按钮讲功能，功能会在最后直接进入应用系统演示；PPT 重点讲项目成果、智能方法和工程结构。

## Slide 3. 项目选题背景：做饭难在连续决策

**页面目的**

说明项目为什么有现实意义，把问题从“不会做菜”扩展为“连续决策成本高”。

**页面核心思想**

用户做饭时最难的不是查到一份菜谱，而是连续推进一整段过程。

**建议页面内容**

标题：

> 项目选题背景：做饭难在连续决策

页面主文案：

- 想不到今天吃什么，选择成本高
- 想到一道菜，却不确定家里食材是否够
- 发现缺料后，不知道该补买、替代还是换菜
- 开始下厨后，还会不断遇到步骤、火候、时间问题
- 做饭过程会变化，需要能根据上下文继续给建议

结论强调：

> 现有菜谱工具擅长展示固定内容，但很难持续陪用户完成任务。

**视觉设计指导**

- 建议用“用户困境漏斗”或“连续卡点时间线”。
- 从左到右展示 5 个卡点：选菜、备料、缺料、烹饪、追问。
- 每个卡点配一个简洁图标，最后汇总成“任务中断”。

**讲述重点**

这里要把问题说得生活化一点：用户不是完全不知道怎么做饭，而是经常在一连串小决策里被卡住。这个问题很适合用智能体方式解决。

## Slide 4. 用户需求：从“想吃什么”到“怎么做出来”

**页面目的**

把背景问题转化为具体系统目标，为后面方案做铺垫。

**页面核心思想**

ChefMate 的目标是帮助用户完成一条完整做饭任务链，而不是提供单点功能。

**建议页面内容**

标题：

> 用户需求：从“想吃什么”到“怎么做出来”

四类核心需求：

- 菜品推荐：结合口味、时间、现有食材、场景给出候选菜
- 食材准备：明确需要哪些食材，判断当前是否能开做
- 缺料处理：列出缺少项，给出补买或调整建议
- 烹饪指导：按步骤推进，支持中途追问和计时提醒

目标表述：

> 让用户用更少的操作，把一次做饭任务从想法推进到执行。

**视觉设计指导**

- 使用“任务闭环圆环图”：需求表达 -> 推荐 -> 备料 -> 烹饪 -> 完成。
- 圆环中心放“对话式智能体”。
- 每个环节下面只放 1 行说明，避免堆字。

**讲述重点**

可以用一个例子串起来：“用户说今晚想吃清淡点，系统推荐；用户拍一张台面，系统识别食材；缺东西就提示补买；备齐后进入步骤指导。”

## Slide 5. 核心洞察：做饭是状态持续变化的任务

**页面目的**

从需求过渡到技术方案，解释为什么需要任务状态和结构化卡片。

**页面核心思想**

要做好这个系统，关键不是让模型多说，而是让系统记住当前任务状态并稳步推进。

**建议页面内容**

标题：

> 核心洞察：做饭是状态持续变化的任务

三点洞察：

- 上下文会持续变化：用户偏好、当前菜品、已备食材、当前步骤都要被记住
- 阶段需要被约束：推荐、备料、烹饪不能随意跳转
- 输出要可操作：结果不只是一段文本，还要能变成推荐卡、备料卡、步骤卡

技术转化：

- 对话负责理解意图
- 任务状态负责记录进度
- 工具调用负责真实修改后端状态
- 前端卡片负责承接用户操作

**视觉设计指导**

- 左侧放“普通聊天”的断裂流程，右侧放“任务智能体”的连续流程。
- 用对比方式展示：文本回复 vs 文本 + 状态 + 卡片 + 数据持久化。
- 可以用双色对比：灰色表示普通问答，绿色表示 ChefMate 任务流。

**讲述重点**

强调项目的技术重点不是“让大模型自由发挥”，而是把大模型放到一个有状态、有工具、有阶段约束的工程系统里。

## Slide 6. 总体方案：任务驱动的烹饪智能体

**页面目的**

展示系统总体解决方案，并对齐后面智能方法与系统实现。

**页面核心思想**

ChefMate 通过阶段状态机和智能体工具调用，把用户对话转化为可执行任务。

**建议页面内容**

标题：

> 总体方案：任务驱动的烹饪智能体

主流程：

- `idea`：接收需求、闲聊、读取用户记忆
- `planning`：推荐菜谱、识别图片食材、选择目标菜品
- `shopping`：展示备料清单、更新食材状态、判断能否开做
- `cooking`：展示步骤指导、推进步骤、完成任务

一句总结：

> 用户自然表达需求，后端将其转换为任务阶段，并通过工具和卡片持续推进。

**视觉设计指导**

- 重点放一张横向状态机图。
- 每个阶段用不同颜色的小块，下面写“输入、处理、输出”。
- 可以在阶段之间标注关键事件：选择菜谱、备齐食材、完成步骤。

**讲述重点**

说明这是系统的骨架。后面讲智能方法时，就围绕“模型如何在这些阶段内调用工具、如何避免乱跳阶段、如何评估效果”展开。

## Slide 7. 智能体上下文设计

**页面目的**

解释 Agent 每轮推理到底依赖哪些上下文信息，让听众理解系统为什么能处理连续任务，而不是只回答单轮问题。

**页面核心思想**

ChefMate 将本轮输入、短期对话、当前任务快照、长期用户记忆和前端卡片状态统一组织为 `AgentTurnContext`，再交给阶段 Prompt 和工具调用流程使用。

**建议页面内容**

标题：

> 智能体上下文设计：让 Agent 知道“当前做到哪一步”

上下文由五类信息组成：

- 本轮输入：用户文本、语音转写、卡片动作、图片附件
- 短期对话：最近消息窗口、会话摘要、当前会话阶段
- 当前任务：任务 ID、任务阶段、目标菜谱、动态菜谱快照
- 用户记忆：口味偏好、忌口过敏、近期历史任务、长期偏好文本
- 前端状态：备料勾选、当前步骤、闪卡焦点、计时器相关状态

设计原则：

- 长期记忆、短期对话、任务状态分层组织，避免混在一段提示词里
- 用结构化任务快照记录食材准备状态和步骤进度
- 将前端卡片状态同步给后端，支持理解“这些都备齐了”“下一步”等省略表达
- 根据当前阶段只开放对应工具，减少模型误调用和阶段乱跳

例子：

> 用户说“下一步是什么”时，Agent 不是只看这句话，而是结合当前任务、当前菜谱、当前步骤和前端卡片状态来回答。

**视觉设计指导**

- 中心放一个大模块：`AgentTurnContext`。
- 周围用五个信息块环绕：本轮输入、短期对话、当前任务、用户记忆、前端状态。
- 从 `AgentTurnContext` 向右连接“阶段 Prompt / 工具白名单 / 结构化卡片输出”。
- 可用小标签标注典型省略表达：`这些都齐了`、`下一步`、`换一道`、`先暂停`。

**讲述重点**

这页要讲清楚项目的“上下文工程”价值：Agent 不是凭一句话猜用户意图，而是每轮都带着任务快照和卡片状态运行，所以才能把做饭这种连续过程接住。

## Slide 8. 智能方法一：阶段型对话 Agent 技术路线

**页面目的**

讲清楚核心智能方法的实现路线，让老师和同学看到技术完整性。

**页面核心思想**

对话 Agent 采用“大模型 + Prompt + ReAct + LangGraph + 任务状态管理”的组合。

**建议页面内容**

标题：

> 智能方法一：阶段型对话 Agent 技术路线

上下文输入：

- 使用上一页的 `AgentTurnContext`
- 包含本轮输入、任务快照、用户记忆、前端卡片状态
- 为阶段 Prompt 和工具调用提供统一输入

处理流程：

- 构建单轮 `AgentTurnContext`
- 根据当前阶段生成 Prompt 与工具白名单
- 使用 LangGraph 执行 ReAct 工具调用
- 工具真实读写任务、菜谱、消息与用户记忆
- 通过 SSE 返回流式文本与结构化卡片

输出结果：

- 用户可读回复
- 推荐卡 / 菜谱详情卡 / 备料卡 / 烹饪步骤卡
- 更新后的任务阶段与动态菜谱快照

**视觉设计指导**

- 建议使用“输入 -> Agent -> 工具 -> 状态更新 -> 前端卡片”的流水线图。
- 在 Agent 中间标注 LangGraph / ReAct / Stage Tools。
- 工具侧可画成 4 组：记忆工具、推荐工具、备料工具、烹饪工具。

**讲述重点**

重点讲“阶段工具白名单”。不同阶段只能调用适合该阶段的工具，例如备料阶段才能推进烹饪，而且后端还会校验必需食材是否 ready。

## Slide 9. Agent 效果评估：可运行、可演示、可迭代

**页面目的**

用评测数据证明智能方法不是只停留在设计层面，而是已经做了工程验证。

**页面核心思想**

基于 58 条手工场景评测，Agent 主链路具备较好的阶段能力和工程稳定性。

**建议页面内容**

标题：

> Agent 效果评估：可运行、可演示、可迭代

评测集：

- 数据集：`agent_core_v1`
- 总样本量：58 条
- 阶段能力测试：28 条
- 状态流转与决策测试：18 条
- 端到端任务完成测试：12 条

关键结果：

- 样例总通过率：86.21%
- 必需工具命中率：96.55%
- 禁止工具违规率：0.00%
- 工具顺序符合率：96.55%
- 最终阶段正确率：87.93%
- 任务状态断言通过率：100.00%
- 端到端样例完成率：83.33%
- 无未预期异常运行率：100.00%
- LLM 裁判平均分：4.4397 / 5

结论：

> 当前系统已经具备可运行、可演示的主链路能力；主要改进点集中在阶段边界决策稳定性。

**视觉设计指导**

- 使用 3 张指标卡突出：总通过率、工具命中率、无异常运行率。
- 右侧用横向条形图展示 4-5 个核心指标。
- 底部用小号文字说明评测集为手工场景集，尚未覆盖真实用户日志。

**讲述重点**

不要只报数字，要解释数字含义：单阶段表现较稳定，工具调用和状态持久化可靠；失败主要发生在阶段边界，例如推进过快或对前端卡片状态理解不稳。

## Slide 10. 系统开发技术栈

**页面目的**

完整展示项目工程实现使用的主要技术，回应“系统开发技术栈”要求。

**页面核心思想**

系统采用前后端分离架构，后端承载智能体运行时，前端承接对话与结构化卡片交互。

**建议页面内容**

标题：

> 系统开发技术栈

前端：

- Vue 3、TypeScript、Vite
- Vue Router、Pinia
- Fetch、SSE、WebSocket
- Web Audio API、AudioWorklet

后端：

- Python 3.10+
- FastAPI、Uvicorn
- Pydantic Settings
- SQLAlchemy、PyMySQL
- LangChain、LangGraph
- OpenAI 兼容接口、aiohttp

数据与模型：

- MySQL
- 本地上传目录与 StaticFiles
- OpenAI 兼容文本模型
- 图片附件处理与识别接口
- 火山引擎 SAUC WebSocket 语音识别

**视觉设计指导**

- 使用“三层技术栈图”：前端层、后端智能体层、数据与模型层。
- 每层使用同一宽度的模块块，避免列表堆砌。
- 右侧可加小标签：REST、SSE、WebSocket、LLM、ASR、VLM。

**讲述重点**

重点不是背技术名词，而是说明每类技术解决什么问题：Vue 负责交互，FastAPI 负责接口和智能体运行时，LangGraph 负责工具调用编排，MySQL 负责任务和会话持久化。

## Slide 12. 项目总结：从普通厨房软件到烹饪智能体

**页面目的**

总结智能体相对普通厨房软件的真实提升点，说明 ChefMate 的价值不是“多了一个聊天入口”，而是改变了用户完成做饭任务的方式。

**页面核心思想**

普通厨房软件更像“菜谱资料库 + 工具集合”，用户仍然要自己搜索、判断、整理和推进；ChefMate 则把这些分散操作组织成一个由智能体协助推进的连续任务。

**建议页面内容**

标题：

> 项目总结：从普通厨房软件到烹饪智能体

普通厨房软件的局限：

- 用户自己找菜谱、筛选、判断和推进
- 信息是静态的，过程上下文容易丢失
- 看得到菜谱，但还要自己判断能不能做、下一步做什么

ChefMate 智能体带来的提升：

- 目标表达：用户说出需求，系统判断该进入哪个阶段
- 上下文连续：记住菜品、食材、步骤和用户偏好
- 任务协作：把推荐、备料、烹饪指导串成可执行流程

收束句：

> 普通厨房软件解决“去哪找信息”，ChefMate 进一步解决“怎样把这顿饭做出来”。

**视觉设计指导**

- 使用左右对比布局：左侧“普通厨房软件”，右侧“ChefMate 智能体”。
- 中间用箭头表示升级路径：信息检索 -> 任务协作。
- 右侧重点突出 3 个价值标签：目标表达、上下文连续、任务协作。
- 底部只放一句加粗结论，避免文字过密。

**讲述重点**

页面文字保持简洁，讲述时再展开：普通厨房软件主要解决信息查找，ChefMate 则把用户原本要自己完成的搜索、判断、记录和推进，放进智能体上下文中协助完成。

## 3. 现场演示推荐脚本

### 演示目标

让观众看到 ChefMate 的核心闭环：自然语言提出需求 -> 智能推荐 -> 选择菜谱 -> 备料检查 -> 烹饪指导。

### 推荐演示话术与动作

1. 直接打开已登录的聊天工作台。
   - 话术：老师不要求展示登录流程，所以我直接从主工作台进入，用一个真实厨房场景演示。

2. 输入需求。
   - 示例：`晚上想吃简单一点的，家里好像有鸡蛋和番茄，帮我推荐一道菜。`
   - 展示点：系统流式回复、推荐卡出现。

3. 查看推荐卡并选择一道菜。
   - 话术：系统不是只返回一段文字，而是把候选菜谱组织成可操作卡片。
   - 展示点：推荐卡、菜谱详情、选择“想尝试”。

4. 展示备料检查卡。
   - 动作：勾选食材准备状态，点击“这些都备齐了”。
   - 展示点：前端卡片状态会回传后端，后端校验后进入烹饪。

5. 展示烹饪步骤卡。
   - 动作：切换列表 / 闪卡模式，展示当前步骤和计时器入口。
   - 展示点：系统不是一次性给长菜谱，而是按步骤指导。

6. 启动一个步骤计时器。
   - 展示点：步骤中的时间信息可以变成前端本地计时提醒。

7. 输入追问：`下一步是什么？`
   - 展示点：系统结合当前任务和步骤状态继续回答，而不是重新开始一轮菜谱问答。

### 演示风险与备选方案

- 如果大模型调用较慢：提前准备一个历史会话，直接恢复到推荐卡或备料卡。
- 如果计时器展示不稳定：只展示步骤卡和当前步骤推进。
- 如果推荐结果生成较慢：提前准备一个历史会话，直接从推荐卡开始演示。
- 如果阶段推进异常：展示后端具备阶段校验和评测结果，并说明这是下一步优化重点。
