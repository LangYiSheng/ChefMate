# ChefMate 智能方法技术报告

## 1. 项目当前智能方法概况

ChefMate 当前涉及两类智能方法：

1. 对话理解与任务推进
2. 食材图像识别

当前进展概况如下：

| 智能方法 | 当前状态 | 说明 |
| --- | --- | --- |
| 对话理解与任务推进 | 已进入主线开发 | 已基于 `FastAPI + LangChain + LangGraph` 实现多阶段任务推进、工具调用、语音识别、ASR 模拟唤醒与结构化卡片联动 |
| 食材图像识别 | 双路线均有成果 | 已接入多模态大模型识别路线，同时保留并完成了 YOLO 路线的阶段性训练与评测 |

## 2. 智能方法一：对话理解与任务推进

### 2.1 当前实现功能

该方法是 ChefMate 的核心智能方法，目标是让系统不只是“聊天”，而是围绕一次真实做饭任务完成状态推进。

当前已经实现的能力包括：

- 识别用户当前意图处于“闲聊、推荐菜品、备料、烹饪”中的哪一类
- 创建活动任务并维护任务阶段
- 通过工具更新用户记忆、任务状态、菜谱步骤与食材状态
- 通过 LangGraph 执行 ReAct 流程
- 返回消息并附带结构化卡片
- 使用豆包流式语音识别模型 2.0 实现语音识别
- 基于 ASR 结果实现模拟唤醒，使用户可以更自然地进入与 ChefMate 的交互

系统的任务主要工作流为：
idea -> planning -> shopping -> cooking -> idea

### 2.2 当前实现方式

该方法当前采用“大模型 API + Prompt + ReAct + LangGraph + 任务状态管理”的组合实现。

其中：

- `LangChain` 用于模型调用、Prompt 模板与工具抽象
- `LangGraph` 用于工作流编排
- `FastAPI` 用于消息接口
- `MySQL` 用于持久化用户、对话、任务、消息
- 豆包流式语音识别模型 2.0 用于语音流式转写

在语音交互部分，当前实现方式为：

1. 用户输入语音
2. 使用豆包流式语音识别模型 2.0 将语音内容实时转写为文本
3. 基于转写文本执行 ASR 模拟唤醒判断
4. 命中唤醒后，把文本结果送入当前智能体主链路继续处理

这样可以让用户在使用智能体时不必完全依赖手动输入，从而让交互更加自然，并进一步提升整体使用体验。

### 2.3 输入数据与输出结果

该智能方法的输入并不是单一的一句用户文本，而是一次完整的“当前轮上下文”。当前实际输入主要包括：

- 用户本轮文本消息
- 用户本轮语音输入转写文本
- 用户本轮卡片动作
- 用户上传图片及其文件链接
- `client_card_state` 中同步上来的前端当前卡片状态
- 当前对话最近消息窗口与压缩摘要
- 当前任务快照，包括菜谱、食材状态、步骤状态、当前阶段
- 用户长期记忆，包括偏好标签、偏好文本、近期历史任务

其输出同样是多层的，既包含直接给用户看的回复，也包含给系统状态机消费的结果：

- 面向前端的流式 Markdown 回复
- 可选的一张结构化卡片
- 更新后的对话阶段、任务阶段与任务快照
- 过程中发生的工具调用轨迹

从产品角度看，该方法的典型输入输出关系可以概括为：

| 输入类型 | 典型内容 | 输出结果 |
| --- | --- | --- |
| 语音输入 | “ChefMate，我想做点清淡的晚饭” | 流式语音转写 + ASR 模拟唤醒 + 进入正常智能体处理流程 |
| 文本闲聊 | “你知道我平时喜欢吃什么吗” | 自然语言回复，可选更新用户记忆 |
| 菜品推荐请求 | “给我推荐点晚餐” | 推荐回复 + 推荐卡片 + 进入 `planning` |
| 备料推进 | “这些都备齐了” | 更新食材状态 + 推进到 `shopping` 或 `cooking` |
| 烹饪追问 | “下一步是什么” | 围绕当前任务步骤进行解释，并同步烹饪步骤卡 |
| 完成/取消 | “做好了” / “今天先不做了” | 完成或取消任务，并回到 `idea` |

### 2.4 智能方法评估数据集

为了验证智能体是否真的具备“阶段理解、工具调用、状态流转和任务完成”能力，我们构建了一套最小可用的评测系统，并基于它整理了一份面向当前主链路的手工场景评估数据集 `agent_core_v1`。

需要说明的是，由于项目时间有限，这一版数据集只完成了最小基础数据集，目标是先覆盖最核心的业务链路，而不是追求大规模样本量。当前数据集并不是用户真实日志回放，而是结合产品需求、阶段设计和已实现工具能力手工编写的场景集。

数据集构建方式如下：

- 直接复用真实后端 `conversation_service`、仓储层与 LangGraph 工作流执行
- 每条样例都显式定义初始状态、用户轮次输入和最终断言
- 初始状态可注入长期记忆、当前阶段、当前任务快照、最近消息、`client_card_state`
- 轮次输入可包含文本、卡片动作、图片附件和局部断言

当前数据集总规模为 58 条样例，包含三类核心评测：

| 数据类别 | 数量 | 主要目标 |
| --- | --- | --- |
| 阶段能力测试 `stage_capability` | 28 | 检查 Agent 在单阶段内能否正确使用工具并完成本阶段目标 |
| 状态流转与决策测试 `transition_decision` | 18 | 检查阶段迁移是否合法、工具顺序是否合理、失败后是否能纠偏 |
| 端到端任务完成测试 `end_to_end` | 12 | 检查从用户输入到任务推进、回滚、取消、完成的完整链路 |

在覆盖类型上，这份最小数据集已经尽量纳入了当前主流程中最关键的复杂情况，包括：

- 文本输入与纯对话场景
- 卡片动作触发
- `client_card_state` 同步场景
- 回滚与取消
- 非法状态迁移
- 目录菜谱与模型现写菜谱
- 图片附件参与推荐
- “这一步 / 下一步”这类强依赖当前任务状态的问题

### 2.5 评价方法

当前评测采用“规则判分为主，LLM 补充裁判为辅”的双层评价方式。

第一层是规则判分，它决定样例是否通过。规则判分主要检查：

- 是否命中要求调用的工具
- 是否错误调用了禁止工具
- 工具调用顺序是否符合预期
- 最终阶段是否正确
- 任务状态断言是否成立
- 执行过程中是否出现未预期异常

只要出现禁止工具、顺序错误、最终阶段错误、任务状态不符或运行异常，当前样例就会被判为失败。

第二层是 LLM 补充裁判，它不覆盖规则判分结果，只用于评价自然语言回复质量。当前裁判维度共有 4 项：

- 状态感知：是否真正基于当前任务状态回答
- 指令遵循：是否正确响应用户指令
- 实用帮助性：是否给出可执行的帮助
- 诚实性与约束遵守：失败时是否如实反馈，没有伪造成功

这种双层评价方式的好处在于：

- 规则判分可以约束系统行为正确性
- LLM 补充裁判可以补充语言质量与交互体验维度
- 两者结合后，既能发现“会说但做错”的问题，也能发现“工具对了但回答不够好”的问题

### 2.6 评价指标

当前智能方法一主要采用以下指标：

| 指标 | 含义 |
| --- | --- |
| 样例总通过率 | 整体样例中规则判分通过的比例 |
| 必需工具命中率 | 需要调用的工具被正确命中的比例 |
| 禁止工具违规率 | 本不应调用的工具被误调用的比例 |
| 工具调用顺序符合率 | 工具顺序满足预期约束的比例 |
| 最终阶段正确率 | 对话最终落在目标阶段的比例 |
| 任务状态断言通过率 | 任务快照与状态断言满足预期的比例 |
| 端到端样例完成率 | 端到端类样例整体通过的比例 |
| 无未预期异常运行率 | 运行过程中未出现异常中断的比例 |
| LLM 裁判平均分 | 补充语言质量裁判的平均得分 |

其中，前 8 项更偏向系统行为正确性，最后 1 项更偏向用户可感知的回复质量。

### 2.7 智能方法在数据集上的性能评估

在 `agent_core_v1` 最小基础数据集上的评测结果如下。

总体结果：

| 指标 | 结果 |
| --- | --- |
| 样例总通过率 | 0.8621 |
| 必需工具命中率 | 0.9655 |
| 禁止工具违规率 | 0.0000 |
| 工具调用顺序符合率 | 0.9655 |
| 最终阶段正确率 | 0.8793 |
| 任务状态断言通过率 | 1.0000 |
| 端到端样例完成率 | 0.8333 |
| 无未预期异常运行率 | 1.0000 |
| LLM 裁判平均分 | 4.4397 |

分类型结果如下：

| 数据类别 | 样例总通过率 | 最终阶段正确率 | 无未预期异常运行率 | LLM 裁判平均分 |
| --- | --- | --- | --- | --- |
| `stage_capability` | 0.9286 | 0.9286 | 1.0000 | 4.5000 |
| `transition_decision` | 0.7778 | 0.8333 | 1.0000 | 4.3889 |
| `end_to_end` | 0.8333 | 0.8333 | 1.0000 | 4.3750 |

从这些结果可以看出几个比较明确的结论：

1. 当前智能体在单阶段能力上已经相对稳定。`stage_capability` 的通过率达到 0.9286，说明在“推荐、备料、烹饪”单阶段内，大多数工具调用与回复都已经能围绕阶段目标展开。

2. 当前系统的工程稳定性较好。无未预期异常运行率达到 1.0000，说明在这批样例中没有出现直接运行崩溃的问题；同时任务状态断言通过率为 1.0000，说明后端任务状态校验和持久化链路整体是可靠的。

3. 当前的主要短板仍然在“状态流转与决策”层。`transition_decision` 通过率只有 0.7778，说明 Agent 在阶段边界处还会出现“推进过快”“阶段判断偏激进”或“工具顺序不够稳”的问题。

4. 自然语言质量总体不错，但仍存在“说得像对的，状态其实不对”的现象。LLM 裁判平均分达到 4.4397，说明回复在可读性、帮助性和表达质量上整体较好；但这并不意味着系统行为完全正确，部分失败样例说明模型有时会在语言上表现自然，却在阶段判断或工具调用上偏离预期。

结合失败案例，可以进一步看到当前最典型的问题主要有三类：

- 阶段推进过快，例如在推荐阶段或备料未完成时过早进入下一阶段
- 面对“现写菜谱”这类开放任务时，偶尔会出现菜谱生成与推进衔接不稳的问题
- 对 `client_card_state` 和“已备齐/下一步”这类强状态依赖输入的理解还不够稳定

因此，当前智能方法一可以认为已经具备了可运行、可演示、可继续迭代的基础能力，但从严格评测结果看，它距离“高稳定性、多场景鲁棒”的成熟智能体还有差距。下一阶段应重点加强：

- 阶段切换前的决策约束
- 现写菜谱场景下的任务推进一致性
- 对前端卡片状态和当前任务快照的联合理解能力

## 3. 智能方法二：食材图像识别

### 3.1 当前实现功能

图像识别用于读取用户上传的食材图片，并输出结构化食材列表，供推荐与任务推进使用。

当前实际采用的是两条路线：

1. 多模态大模型识别路线
2. 本地 YOLO 检测路线

### 3.2 输入数据与输出结果

输入样例：

```text
POST /api/files/upload
POST /api/conversations/{id}/messages/stream

用户上传图片后，模型调用 recognize_image_ingredients 工具识别食材
```

输出样例：

```json
{
  "ingredients": [
    {"name": "番茄"},
    {"name": "鸡蛋"},
    {"name": "葱"}
  ]
}
```

### 3.3 训练数据集、规模与格式

对于多模态大模型路线：

- 使用已训练好的多模态大模型 API
- 不单独训练

对于 YOLO 路线：

- 基础数据集：FoodSeg103
- 补充数据：自建食材图片数据
- 当前目标类别：22 类高频常见食材
- FoodSeg103 规模：7,118 张图像，103 类
- 训练时需要筛选、映射到本项目的 22 类目标食材，再转换为检测训练格式

当前 22 类目标食材包括：

- 鸡蛋
- 番茄
- 葱
- 土豆
- 洋葱
- 黄瓜
- 胡萝卜
- 青椒
- 茄子
- 大蒜
- 姜
- 花菜
- 生菜
- 包菜
- 豆腐
- 米饭
- 香菜
- 芹菜
- 豆芽
- 豆角
- 香菇
- 金针菇

### 3.4 评价方法与评价指标

多模态大模型路线主要使用自建测试图片集做人工核验，指标包括：

- Precision
- Recall
- F1
- Jaccard

YOLO 路线主要使用检测模型指标：

- Precision
- Recall
- mAP@50
- mAP@50:95
- Fitness

### 3.5 智能方法在数据集上的性能评估

多模态大模型路线在当前自建测试集上的食材级结果如下：

- TP：12
- FP：2
- FN：0
- Precision：0.8571
- Recall：1.0000
- F1：0.9231
- Jaccard：0.8571

YOLO11n 路线在 FoodSeg103 的 22 类子集验证集上，第 100 轮训练后的结果如下：

- Precision：0.5294
- Recall：0.4026
- mAP@50：0.4412
- mAP@50:95：0.3502
- Fitness：0.3502

当前可以得出的结论是：

- YOLO 路线确实已经完成了训练与阶段性评测，不能视为“没有做”
- 但从当前结果看，YOLO 路线的识别效果仍弱于大模型路线，尤其在召回率和复杂场景稳定性上还有提升空间
- 因此，当前项目阶段更适合把大模型路线作为效果更好的识别方案，同时保留 YOLO 路线作为本地可控、可持续优化的训练方向
