# ChefMate 智能方法技术报告

## 1. 项目当前进展分析

本项目当前涉及三类智能方法：

1. 对话理解与任务推进
2. 结构化推荐系统
3. 食材图像识别

当前进展概况如下：

| 智能方法 | 当前进展状态 | 说明 |
| --- | --- | --- |
| 对话理解与任务推进 | 原型 demo 阶段 | 已有任务管理与工作流骨架，但尚未进入主线开发 |
| 结构化推荐系统 | 已有查询原型 | 已实现按名称、标签、现有食材三条路径查询与推荐，后续计划引入双塔模型 |
| 食材图像识别 | 双路线开发中 | 已完成大模型视觉识别通路，本地 YOLO 训练方案仍在开发 |

## 2. 智能方法一：对话理解与任务推进

### 2.1 实现功能

该方法用于将用户自然语言需求转换为可执行任务，并在任务生命周期中持续推进。当前该部分仍处于原型 demo 阶段，主要承担以下功能：

- 理解用户当前意图属于“推荐菜品、确认菜品、备料检查、烹饪指导”中的哪一类
- 创建任务对象并记录当前阶段
- 支持任务恢复、阶段切换和继续推进
- 将用户输入路由到对应技能模块

### 2.2 输入数据与输出结果

输入数据主要为用户文本消息以及当前任务上下文。

输入样例：

```json
{
  "conversation_id": "c-001",
  "message": "我想做个快一点的晚饭，最好清淡一点。"
}
```

输出结果为任务快照与下一步动作结果。

输出样例：

```json
{
  "task": {
    "id": "t-001",
    "conversation_id": "c-001",
    "stage": "RECOMMEND",
    "status": "RUNNING",
    "current_node": "recommend",
    "semantic_query": {
      "raw_text": "我想做个快一点的晚饭，最好清淡一点。",
      "target_recipe_name": null,
      "available_ingredients": [],
      "health_goal": null,
      "max_cook_time": null,
      "available_tools": []
    }
  },
  "assistant_message": "已生成候选菜品，请确认目标菜品。"
}
```

### 2.3 选用的模型或 API

该方法当前采用“大模型 API + Prompt 约束 + 工作流控制 + 任务状态管理”的组合实现方案，具体包括：

- 大模型 API：负责自然语言理解、ReAct 推理和补充解释
- Prompt 工程：约束输出格式和任务边界
- 工作流骨架：当前使用自定义工作流入口，后续计划接入 LangGraph
- 任务管理器：负责记录任务阶段与状态

### 2.4 训练数据集、规模与格式

该方法当前不采用自训练模型，因此无独立训练数据集。其能力主要来自：

- 已训练好的大模型 API
- 结构化任务对象
- 固定工作流与技能调用规则

### 2.5 测试数据集

当前尚未建立正式的任务推进测试数据集。现阶段的测试方式主要是接口级联通验证，后续计划构建一组人工任务样例，覆盖：

- 模糊需求推荐
- 已有明确菜品
- 基于现有食材推荐
- 任务中断恢复
- 烹饪中途追问

### 2.6 模型性能

当前尚未形成可重复的量化性能指标。目前已完成：

- 会话创建
- 任务创建与恢复
- 基本阶段路由

尚待完成的内容包括：

- 基于标准任务集的阶段推进准确率评测
- 中断恢复成功率统计
- LangGraph 工作流稳定性测试

## 3. 智能方法二：菜谱推荐系统

### 3.1 实现功能

推荐系统用于根据用户长期画像和即时需求，为用户返回候选菜品。当前已实现查询型原型，包含以下能力：

- 菜谱名称检索
- 基于固定标签的菜谱过滤
- 基于已有食材的可做菜品检索
- 推荐解释所需的结构化结果输出

当前推荐系统仍停留在普通查询和规则过滤阶段，不依赖训练式推荐模型。后续计划引入基于嵌入训练的双塔模型，与现有标签筛选逻辑组成混合推荐方案。

### 3.2 输入数据与输出结果

推荐系统的输入来自三类信息：

- 用户长期画像
- 当前会话文本需求
- 已识别的现有食材

输入样例一：文本推荐

```json
{
  "user_profile": {
    "flavor_preferences": ["清淡"],
    "health_goal": "减脂友好",
    "available_tools": ["炒锅"],
    "max_cook_time": 20
  },
  "query": "我想吃个清淡一点的晚饭，二十分钟内能做好。"
}
```

输出样例一：

```json
{
  "query": "我想吃个清淡一点的晚饭，二十分钟内能做好。",
  "route": "tag_filter",
  "filters": {
    "health": ["清淡饮食"],
    "time": ["10到20分钟"]
  },
  "candidates": [
    {
      "id": 1,
      "name": "西红柿炒鸡蛋",
      "difficulty": "简单",
      "estimated_minutes": 15,
      "score": 0.92
    }
  ]
}
```

输入样例二：基于现有食材推荐

```json
{
  "ingredients": ["番茄", "鸡蛋", "葱"]
}
```

输出样例二：

```json
{
  "ingredients": ["番茄", "鸡蛋", "葱"],
  "route": "ingredient_search",
  "candidates": [
    {
      "id": 1,
      "name": "西红柿炒鸡蛋",
      "estimated_minutes": 15,
      "score": 0.95,
      "exact_makeable": true,
      "matched_ingredients": ["番茄", "鸡蛋"],
      "missing_required_ingredients": []
    }
  ]
}
```

### 3.3 选用的模型或 API

推荐系统当前采用的是**结构化查询与规则推荐方法**。

具体方法包括：

- 菜名检索：名称相似度匹配
- 标签过滤：固定标签字典匹配
- 食材推荐：必选食材匹配率与缺失量计算
- 排序：规则过滤后进行多因素打分

后续计划采用的智能方法为双塔推荐模型，其思路为：

- 用户塔：编码用户长期画像与即时需求
- 菜谱塔：编码结构化菜谱特征
- 通过向量相似度完成召回，再与现有标签筛选结果融合

后续推荐系统计划采用“标签过滤 + 双塔召回/排序”混合方案。

### 3.4 训练数据集、规模与格式

该方法当前不涉及模型训练，主要依赖结构化菜谱数据库。

当前数据来源为：

- `HowToCook` 仓库作为基础菜谱来源
- 基于 AI 标注与人工筛查完成结构化数据创建和整理
- 固定标签字典

数据规模与格式：

- 规模：数百条高质量中文菜谱
- 存储格式：MySQL 结构化表
- 核心表：
  - `recipe`
  - `recipe_ingredient`
  - `recipe_step`
  - `recipe_tag_category`
  - `recipe_tag`
  - `recipe_tag_map`

### 3.5 测试数据集

当前尚未建立正式的推荐测试集。后续计划构建一套人工标注测试集，至少包含以下三类查询：

- 菜名检索类
- 标签过滤类
- 食材匹配类

每条测试样例计划包含：

- 输入查询
- 期望 Top-1 菜谱
- 期望 Top-3 候选集合
- 允许的替代答案

### 3.6 模型性能

当前推荐系统已经具备以下能力：

- 可按名称搜索菜谱
- 可按标签组合过滤菜谱
- 可按现有食材查找可做菜品

但当前尚未形成正式性能指标，因此以下结果暂未给出：

- Top-1 命中率
- Top-3 命中率
- Recall@K
- MRR

当前推荐系统已完成查询原型与数据联通，但尚未进入训练型推荐阶段。

## 4. 智能方法三：食材图像识别

### 4.1 实现功能

食材图像识别用于读取用户上传的食材图片，并输出可用于做菜决策的食材列表。当前采用两条并行路线：

1. 利用多模态大模型进行图像识别
2. 利用本地训练的 YOLO 模型进行识别

两条路线的共同目标包括：

- 识别台面摆放食材的主要种类
- 过滤器皿、包装袋等无关对象
- 返回结构化食材列表
- 为后续推荐系统提供输入

### 4.2 输入数据与输出结果

当前接口接收图片文件和可选的文字提示。

输入样例：

```text
POST /api/vision/ingredients/detect
Content-Type: multipart/form-data

image=<binary image>
user_hint=这些都是今晚准备做饭的食材
```

输出样例：

```json
{
  "backend": "llm",
  "model": "gpt-4.1-mini",
  "ingredients": [
    {"name": "番茄"},
    {"name": "鸡蛋"},
    {"name": "葱"}
  ],
  "raw_text": "{\"ingredients\":[\"番茄\",\"鸡蛋\",\"葱\"]}"
}
```

### 4.3 选用的模型或 API

图像识别方法采用双路线方案：

1. **路线一：多模态大模型识别**
   当前已完成接口联调，使用多模态大模型 API 直接识别图片中的食材。

   该路线当前存在的主要问题是：模型返回的食材称呼与菜谱数据库中的食材命名不完全统一，容易影响后续推荐与备料判断。因此，后续开发方向是通过提示词优化、术语映射，并进一步探索针对食材命名一致性的微调方法，使输出更贴近菜谱库中的标准食材名称。

2. **路线二：本地 YOLO11n 模型识别**
   该路线计划通过本地训练 YOLO11n 模型识别台面摆放食材，作为更可控、更易评测的正式方案。目前仍在开发中，尚未接入主流程。

   当前识别范围只聚焦于 22 类高频、常见且与现有菜谱关联度较高的食材，分别为：鸡蛋、番茄、葱、土豆、洋葱、黄瓜、胡萝卜、青椒、茄子、大蒜、姜、花菜、生菜、包菜、豆腐、米饭、香菜、芹菜、豆芽、豆角、香菇、金针菇。

### 4.4 训练数据集、规模与格式

对于路线一：

- 调用已训练好的多模态大模型 API
- 不单独使用训练数据集

对于路线二：

- 基础数据集：FoodSeg103
- 补充数据：自建食材图片数据
- 识别类别范围：当前先针对 22 类食材进行训练与测试，不直接覆盖全部食品类别
- 数据集规模：
  - FoodSeg103：7,118 张图像，103 类
  - 自建训练数据：当前尚未完成采集
- 数据格式：
  - FoodSeg103 原始为食物图像标注数据
  - 后续训练 YOLO 时需筛选并映射至上述 22 类目标食材，再转换为检测训练格式

### 4.5 测试数据集

图像识别当前按两条路线分别使用不同测试数据：

1. 多模态大模型识别路线：使用项目自建测试集进行验证，当前已准备约 40 张测试图片，其中桌面摆放食材图片与冰箱内放置食材图片各约占一半。该测试集重点覆盖多食材共同摆放、厨房台面常见背景、遮挡与弱光情况。

2. 本地 YOLO11n 识别路线：该路线的成绩计算基于 FoodSeg103 提供的测试集，在当前收敛后的 22 类目标食材范围内进行评测。目前已获得阶段性实验结果，后续仍将继续补充训练轮次和优化结果。

后续计划基于自建测试图片持续进行人工核验，并统计大模型识别结果与菜谱标准食材命名的一致率。

### 4.6 模型性能

当前已获得多模态大模型路线在自建测试集上的初步结果，本地 YOLO11n 路线也已完成阶段性实验，但该路线尚未完成系统接入与完整优化，因此整体视觉模块仍处于阶段性评测状态。

其中，多模态大模型在当前测试数据上的食材级识别结果如下：

- TP：12
- FP：2
- FN：0
- Precision：0.8571
- Recall：1.0000
- F1：0.9231
- Jaccard：0.8571

上述结果说明，当前大模型路线在测试样例中具有较高召回率，能够较完整地识别图片中的目标食材，但仍存在少量误识别情况，主要问题仍然集中在食材命名与菜谱标准名称的一致性上。

本地 YOLO11n 路线在 FoodSeg103 的 22 类子集验证集上也已获得阶段性实验结果。在第 100 轮训练后，模型性能如下：

- Precision：0.5294
- Recall：0.4026
- mAP@50：0.4412
- mAP@50:95：0.3502
- Fitness：0.3502

上述结果表明，模型已经能够较稳定地学习到目标食材特征，mAP 指标较前期实验已有明显提升，但召回率和复杂场景下的识别效果仍有进一步优化空间。

当前尚不能形成完整视觉性能结论，原因如下：

- 本地 YOLO11n 路线当前结果仍为阶段性结果
- 大模型路线使用的自建测试集规模仍较小
- 两条路线当前使用的测试数据来源不同
- 仍需继续补充更大规模测试与分类别统计

## 5. 当前报告结论

1. 对话理解与任务推进目前仍处于原型 demo 阶段，尚未进入主线开发。
2. 推荐系统已完成结构化查询原型，后续将引入双塔模型形成混合推荐方案。
3. 图像识别采用“大模型识别 + 本地 YOLO11n 识别”双路线，其中大模型路线已得到自建测试集上的初步结果，YOLO11n 路线也已获得 FoodSeg103 22 类子集验证集上的阶段性实验结果。
