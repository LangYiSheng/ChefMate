ChefMate 采用“前端表示层—接口接入层—会话与智能体编排层—业务服务层—数据与基础设施层”的分层架构。系统前端以类似 ChatGPT 的响应式 Web 页面为载体，支持 PC 与移动端访问。前端所有请求统一进入 Spring Boot 后端，由其承担用户管理、会话管理、阶段状态控制、提示词编排、工具路由、卡片状态维护和响应聚合等核心职责。

在业务能力上，系统将推荐、菜谱管理、食材管理、烹饪流程、健康记录、用户画像等功能划分为独立业务服务。其中推荐系统采用 Python 实现，用于承载推荐算法训练与在线推理；Spring Boot 通过接口方式调用该服务，并将推荐结果与当前会话上下文结合后交由大模型生成自然语言说明与结构化卡片。

系统智能体采用阶段化工作机制，在普通对话、菜品选择、食材准备、厨房烹饪、已完成等不同阶段加载不同提示词与工具集合，并通过“单活动卡片”机制统一管理当前对话中的结构化交互内容。底层采用 MySQL、对象存储以及向量数据库分别承载结构化数据、缓存状态、图片文件与语义检索能力，从而支撑整个系统的对话式烹饪辅助闭环。

本系统的智能体由各种工具和提示词组成，提示词和工具会在用户的不同阶段不断调整。
大致分为：普通对话（基本，用于一般聊天，或者特殊任务，比如上传菜谱、记录食材），菜品选择阶段、食材准备阶段、厨房烹饪阶段、已完成（关闭对话，不可继续）。
用户的输入为文字或图片。
智能体的输出为文字和卡片。 卡片实际上是一段结构化数据，在一个对话中，卡片会不断变化，但是只会有一个卡片显示在用户面前。智能体可以按照用户的要求更新和修改卡片。
卡片显示类型包含：菜品选择卡片（是一个选择页面，选后自动发送消息"我想吃..."，也可以换一批）、食材准备卡片（最开始是一个目录的样子，显示需要/缺少的食材的名称和份量，然后点进其中一个就会进入类似闪卡一样的详情，用户可以打勾、上一个、下一个，这样的操作）、烹饪流程卡片（类似食材准备卡片，先展示流程步骤，然后一步步展示类似于闪卡的下一步下一步...可选的配图）

菜谱在数据库中也需要存储一段结构化数据JSON，用于展示烹饪流程卡片。支持每一步可选的配图。

菜谱展示卡片（用于用户上传菜谱后，展示给用户识别的菜谱是否正确，也可以进一步修改。也可以用于菜谱选择后给用户二次确认。）

系统的重点开发在于智能体流程、工具的开发，其中工具中重点在于推荐系统工具。 在不同的阶段，系统可以给用户提供不同的工具和提示词。 
智能体在流程中可能需要的工具有：
阶段管理工具（可以让智能体更新阶段信息，获取进一步的工具）
卡片修改工具（更新各种卡片的内容）
菜谱推荐展示工具（调用推荐系统获取推荐的菜品 并创建卡片）
食材准备展示工具（提交需要准备的食材 并创建卡片）
烹饪流程展示工具（输出某个菜谱 并创建卡片）
创建菜谱工具（用于创建一个菜谱，并创建菜谱展示卡片，经过用户同意后就可以以用户身份上传一个菜谱）
食材管理系列工具（如果食材管理启用的话，用户可以通过普通聊天来管理食材）

注意 对话始终仅维护一个卡片，创建新卡片会覆盖旧卡片的内容。
