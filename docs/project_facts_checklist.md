# ChefMate 项目事实清单

## 1. 文档用途

这份文档不是最终报告，而是为后续两份文档做准备的“统一底稿”：

- 概要设计报告
- 智能方法调研报告

它的作用是先把项目中已经明确的事实、还未明确但必须确认的事实、以及写报告前需要补齐的材料整理出来。等这些内容达成一致后，再正式撰写两份报告，会更稳定，也能避免后面反复推翻。

建议后续讨论时，直接围绕本清单逐项确认。

状态说明：

- `已确认`：目前已有较稳定共识，可直接进入后续报告
- `待确认`：方向大致明确，但还需要我们拍板
- `待调研`：必须查资料或补材料后才能写得扎实

| 事实项 | 当前结论 | 状态 |

1. `flavor_preferences`：口味偏好
2. `dietary_restrictions`：忌口/过敏
3. `health_goal`：健康目标
4. `cooking_skill_level`：做饭熟练度
5. `max_cook_time`：可接受做饭时长
6. `available_tools`：常用厨具

## 6. 建议的讨论顺序

为了避免一上来就陷进技术细节，建议按下面顺序逐项确认：

1. 先确认项目边界：到底做什么，不做什么
2. 再确认功能结构：一级功能、二级功能、主流程、支撑功能
3. 再确认技术路线：前端、后端、模型、图片能力、推荐能力怎么组合
4. 再确认数据与接口：核心数据对象有哪些，要不要细到字段级
5. 最后确认智能方法调研范围：调研哪些模型、哪些数据集、比较哪些技术路线

## 7. 下一步最值得优先拍板的问题

如果我们想尽快进入正式写报告阶段，目前还最值得优先补齐的是下面这些内容：

1. 推荐模块的“多因素打分公式”准备写到多细
2. 用户偏好、会话、消息、卡片、阶段这几张核心表准备写到多细
3. 简单接口清单准备列到哪些接口
4. 自建中文菜谱数据集的样本规模最终写多少条更稳妥
5. 推荐数据中的标签字段是否还需要补充其他字段
6. YOLO 方案的具体版本和训练方式是否需要进一步明确
7. 图像识别报告中是否要加入“只支持摆放食材图像，不支持冰箱场景”的限制说明
8. 大模型 API 在调研报告中准备选择哪些候选模型作为主选/备选
9. LangGraph 的使用范围要写到“框架层面”还是“具体工作流实现层面”
10. 是否需要在报告中补充少量示例 Prompt 或工作流节点示意图
