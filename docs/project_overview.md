# ChefMate 项目简介

**ChefMate** 是一个面向日常做饭场景的智能烹饪助手。它的核心价值不在于单独提供菜谱，而在于陪用户完成一整段更真实的任务流程：先帮用户决定吃什么，再帮用户确认需要准备什么，随后结合用户当前已有食材判断能不能做、还缺什么，最后继续提供做饭指导。

相比传统菜谱工具，ChefMate 更强调“任务推进”而不是“信息堆叠”。用户不必先把所有条件整理得很完整，也不必在多个页面之间自己切换流程，而是可以直接通过对话表达当前想法，让系统一步一步带着往下走。

## 系统核心定位

ChefMate 的定位是一个围绕做饭任务持续陪伴用户的智能助手。

它需要解决的不是单个孤立问题，而是把下面这些问题连起来：

- 今天吃什么更合适
- 这道菜需要准备什么
- 我现在手头这些食材够不够
- 如果不够，还需要买什么，或者能不能换个方案
- 材料齐了之后，这道菜具体怎么做

因此，ChefMate 不是“菜谱展示页 + 若干功能按钮”的简单组合，而是一个能够理解上下文、衔接不同阶段、持续帮助用户完成任务的系统。

## 当前聚焦范围

现阶段，项目先聚焦最核心的主流程，不把范围铺得过大。当前重点不放在以下方向：

- 不以“长期食材台账管理”作为重点功能
- 暂不重点展示“分享自制菜谱”
- 暂不重点展示“记录饮食情况”

这些内容未来可以继续扩展，但不属于当前阶段最核心、最需要优先完成的体验。

当前版本更关心的是：当用户想吃一顿饭时，系统能不能真正把这顿饭从“想法”推进到“做成”。

## 核心流程

ChefMate 当前最典型的流程可以概括为：

1. 推荐吃什么
2. 确定吃什么
3. 明确这道菜需要哪些食材
4. 通过拍照或描述识别用户现有食材
5. 判断食材是否备齐
6. 如果未备齐，继续指导用户补买
7. 如果已备齐，进入烹饪指导

这个流程并不意味着所有用户都必须严格按同一顺序使用。它只是一个最能体现产品价值的主线场景。现实中，用户的起点可能不同：

- 有的用户先问“今晚吃什么”
- 有的用户先说“我就想吃这个”
- 有的用户会先拍一张冰箱照片，让系统判断能做什么
- 有的用户已经准备得差不多了，只想要后续的做饭指导

所以，ChefMate 的目标不是把用户强行塞进固定步骤里，而是让系统能够识别用户此刻处在哪个场景，并自动衔接到合适的下一步。

## 典型场景说明

### 场景一：不知道吃什么

用户只知道自己今天想吃得清淡一点，或者想做得快一点，但没有明确目标菜品。系统需要先理解这些模糊需求，再给出更符合当前情况的建议，并帮助用户确认最终要做的菜。

### 场景二：已经想好一道菜

用户已经有明确目标，例如“今晚想做番茄炒蛋”。这时系统不需要重新推荐，而是应立刻进入准备确认，帮助用户看清楚这道菜需要什么、现在缺什么。

### 场景三：靠图片快速确认现状

用户不想逐项描述家里有哪些食材，而是更愿意拍一张冰箱、购物袋或台面照片。系统应能够基于图片理解大致情况，并结合目标菜品继续判断是否能做、还差什么。

### 场景四：边做边问

用户真正开始做饭后，往往会出现很多临时问题，例如“这一步是不是该翻面了”“我没有某种调料怎么办”。系统要做的不是把用户送到一份静态菜谱前就结束，而是继续在当前上下文里提供帮助。

## 产品思路

ChefMate 的产品思路可以概括为两点：

- 以用户场景为中心，而不是以功能页面为中心
- 让模型在工具支持和提示词引导下，尽量自主应对大多数真实做饭场景

也就是说，我们在开发时不是单纯拼接几个割裂的功能，而是希望通过合理的能力组织方式，让系统能够围绕用户当前任务，自动判断下一步该提供什么帮助。这样，用户感受到的将不是“我在操作很多功能”，而是“有一个助手在陪我把这顿饭做出来”。

## 交互方式

ChefMate 的交互应尽量贴近日常对话，同时在关键节点给出清晰结果。用户可以通过文字或图片表达需求，系统则在不同阶段输出适合当前任务的信息，例如：

- 推荐候选菜品
- 所需食材与缺失食材
- 补买建议或替代建议
- 分步骤烹饪指导

整体上，ChefMate 希望带给用户的体验是：少一点来回切换和反复确认，多一点顺着场景自然往下走的连贯感。
